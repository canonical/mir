# Rust API for Mir (miral-rs)

## Problem Statement

The primary API for compositor authors is `miral` (C++). To extend Mir's longevity and accessibility, we need an idiomatic Rust API that provides the same functionality. The API should feel natural to Rust developers — not like a thin C++ wrapper — while leveraging the existing `miral` implementation via `cxx.rs` bindings.

## Proposed Approach

### Architecture: Layered Design

```
┌─────────────────────────────────────────────┐
│  miral  (idiomatic Rust public API crate)   │  ← Compositor authors use this
├─────────────────────────────────────────────┤
│  miral-sys  (cxx.rs FFI + bindgen enums)    │  ← Thin unsafe bridge to C++
├─────────────────────────────────────────────┤
│  miral  (existing C++ library)              │  ← Proven, battle-tested implementation
└─────────────────────────────────────────────┘
```

**Two crates:**

1. **`miral-sys`** — Low-level FFI bindings. Uses **bindgen** for the C-linkage enums (from `mir_toolkit/common.h` and `mir_toolkit/events/enums.h`) and **cxx.rs** for the C++ class bindings (MirRunner, WindowManagerTools, etc.).
1. **`miral`** (the Rust crate) — Idiomatic, safe Rust API built on top of `miral-sys`. Uses Rust traits, builders, enums, and ownership semantics.

### Enum Strategy: bindgen for C enums

The Mir toolkit enums (`MirWindowState`, `MirWindowType`, `MirResizeEdge`, `MirInputEventModifier`, etc.) are declared in pure C headers:

- `include/core/mir_toolkit/common.h` — 26 `typedef enum` declarations
- `include/core/mir_toolkit/events/enums.h` — 11 `typedef enum` declarations

These are ideal candidates for **bindgen** because:

- They use C linkage (no C++ name mangling)
- They have explicit integer values and `#repr(C)` semantics
- bindgen will auto-generate the correct `repr(u32)`/`repr(i32)` types
- It avoids the manual maintenance burden of keeping 37+ enums in sync

The `miral-sys` `build.rs` will:

1. Run `bindgen` on `common.h` and `events/enums.h` → generates raw enum constants
1. Run `cxx-build` for the C++ class bridge definitions

The idiomatic `miral` crate then wraps the raw bindgen constants in proper Rust enums with `From`/`TryFrom` conversions:

```rust
// miral-sys (generated by bindgen)
pub const mir_window_state_restored: u32 = 1;
pub const mir_window_state_minimized: u32 = 2;
// ...

// miral (idiomatic layer)
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum WindowState {
    Unknown,
    Restored,
    Minimized,
    Maximized,
    // ...
}

impl From<u32> for WindowState { /* ... */ }
impl From<WindowState> for u32 { /* ... */ }
```

**Note:** The existing `evdev-rs` crate manually duplicates a few enums inside `#[cxx::bridge]` blocks. This was reasonable for the ~5 enums it needed, but would be burdensome for the full 37 enums `miral-rs` requires. bindgen is the right approach here.

### Key Idiomatic Rust Design Decisions

| C++ miral pattern                        | Rust equivalent                                                     |
| ---------------------------------------- | ------------------------------------------------------------------- |
| `WindowManagementPolicy` (virtual class) | `trait WindowManagementPolicy`                                      |
| `WindowManagerTools` (opaque handle)     | `struct WindowManagerTools` with safe methods                       |
| `WindowSpecification` (optional fields)  | Builder pattern with `WindowSpecificationBuilder`                   |
| `MirRunner::run_with({...})`             | `MirRunner::new().add(...).run()` (builder)                         |
| `operator()(mir::Server&)` functors      | `trait ServerExtension` with `fn apply(&self, server: &mut Server)` |
| `mir::optional_value<T>`                 | `Option<T>`                                                         |
| `std::shared_ptr<Workspace>`             | `Arc<Workspace>` (or opaque handle)                                 |
| `ConfigurationOption` (overloaded ctors) | `ConfigurationOption` with `ConfigValue` sealed trait + methods     |
| C-linkage Mir enums                      | bindgen + idiomatic Rust enum wrappers                              |
| Callbacks (`std::function`)              | `Fn`/`FnMut` trait objects or closures                              |
| `WindowInfo` (mutable reference)         | `&mut WindowInfo` with getter/setter methods                        |
| Error handling (exceptions)              | `Result<T, MirError>`                                               |

### WindowManagementPolicy Trait (Final Design)

The trait uses a hybrid approach:

- **Individual methods** for `handle_*` / `place_*` / `confirm_*` — these have distinct return types
- **Single `advise` method** with a `#[non_exhaustive]` event enum for notifications — these are all fire-and-forget

```rust
pub trait WindowManagementPolicy: Send + 'static {
    /// Called to determine where a new window should be placed.
    /// Default: honors the requested specification as-is.
    fn place_new_window(
        &mut self,
        _app_info: &ApplicationInfo,
        requested: &WindowSpecification,
    ) -> WindowSpecification {
        requested.clone()
    }

    /// Called when a window's first buffer has been posted.
    /// Default: no-op.
    fn handle_window_ready(&mut self, _window_info: &mut WindowInfo) {}

    /// Called when a client requests modifications to its window.
    /// Default: applies all requested modifications.
    fn handle_modify_window(
        &mut self,
        window_info: &mut WindowInfo,
        modifications: &WindowSpecification,
    ) {
        self.tools().modify_window(window_info, modifications);
    }

    /// Called when a client requests its window be raised.
    /// Default: raises the window tree.
    fn handle_raise_window(&mut self, window_info: &mut WindowInfo) {
        self.tools().raise_tree(&window_info.window());
    }

    /// Confirm placement of a maximized/fullscreen window.
    /// Default: accepts the suggested placement.
    fn confirm_placement_on_display(
        &mut self,
        _window_info: &WindowInfo,
        _new_state: WindowState,
        new_placement: &Rectangle,
    ) -> Rectangle {
        *new_placement
    }

    /// Handle a keyboard event. Return true if consumed.
    /// Default: not consumed.
    fn handle_keyboard_event(&mut self, _event: &KeyboardEvent) -> bool { false }

    /// Handle a touch event. Return true if consumed.
    /// Default: not consumed.
    fn handle_touch_event(&mut self, _event: &TouchEvent) -> bool { false }

    /// Handle a pointer event. Return true if consumed.
    /// Default: not consumed.
    fn handle_pointer_event(&mut self, _event: &PointerEvent) -> bool { false }

    /// Handle a client-initiated move request. Default: no-op.
    fn handle_request_move(&mut self, _window_info: &mut WindowInfo, _input_event: &InputEvent) {}

    /// Handle a client-initiated resize request. Default: no-op.
    fn handle_request_resize(
        &mut self,
        _window_info: &mut WindowInfo,
        _input_event: &InputEvent,
        _edge: ResizeEdge,
    ) {}

    /// Confirm child window placement when parent moves.
    /// Default: applies the displacement to current position.
    fn confirm_inherited_move(
        &mut self,
        window_info: &WindowInfo,
        movement: Displacement,
    ) -> Rectangle {
        Rectangle {
            top_left: window_info.window().top_left() + movement,
            size: window_info.window().size(),
        }
    }

    /// Called for all advisory notifications from the compositor.
    /// Default: no-op. Override to react to lifecycle events.
    fn advise(&mut self, _event: Advice) {}

    /// Access to the window manager tools. Required.
    fn tools(&self) -> &WindowManagerTools;
}

/// Advisory notifications from the compositor.
///
/// These are fire-and-forget notifications — no response is expected.
/// Use `match` to handle the events you care about and ignore the rest
/// with a wildcard (`_ => {}`).
///
/// Marked `#[non_exhaustive]` so new events can be added without breaking
/// existing code.
#[non_exhaustive]
pub enum Advice<'a> {
    /// A group of related notifications is starting.
    Begin,
    /// A group of related notifications has ended.
    End,

    // --- Application lifecycle ---
    NewApp { app: &'a mut ApplicationInfo },
    DeleteApp { app: &'a ApplicationInfo },

    // --- Window lifecycle ---
    NewWindow { window_info: &'a WindowInfo },
    DeleteWindow { window_info: &'a WindowInfo },
    FocusGained { window_info: &'a WindowInfo },
    FocusLost { window_info: &'a WindowInfo },
    StateChange { window_info: &'a WindowInfo, state: WindowState },
    MoveTo { window_info: &'a WindowInfo, top_left: Point },
    Resize { window_info: &'a WindowInfo, new_size: Size },
    Raise { windows: &'a [Window] },

    // --- Output lifecycle ---
    OutputCreate { output: &'a Output },
    OutputUpdate { updated: &'a Output, original: &'a Output },
    OutputDelete { output: &'a Output },

    // --- Application zone lifecycle ---
    ZoneCreate { zone: &'a Zone },
    ZoneUpdate { updated: &'a Zone, original: &'a Zone },
    ZoneDelete { zone: &'a Zone },
}
```

### ConfigurationOption

Add user-defined configuration options to Mir's option handling. Options can come from
the command line, environment variables, config files, or defaults. Callbacks fire on
every value change, which may happen multiple times throughout the program's lifetime.

#### Sealed Traits

```rust
mod private { pub trait Sealed {} }

/// Types that can be used as configuration option values.
pub trait ConfigValue: private::Sealed {}
impl ConfigValue for i32 {}
impl ConfigValue for f64 {}
impl ConfigValue for String {}
impl ConfigValue for bool {}

/// Subset of `ConfigValue` that supports optional (no default).
/// C++ `mir::optional_value` exists for int, string, and bool — but not double.
pub trait OptionalConfigValue: ConfigValue {}
impl OptionalConfigValue for i32 {}
impl OptionalConfigValue for String {}
impl OptionalConfigValue for bool {}
```

#### API

```rust
pub struct ConfigurationOption { /* ... */ }

impl ConfigurationOption {
    /// Required value with default. Callback fires on every change.
    pub fn new<T: ConfigValue>(
        name: impl Into<String>,
        description: impl Into<String>,
        default: T,
        callback: impl FnMut(T) + Send + 'static,
    ) -> Self { /* ... */ }

    /// Optional value (no default). Callback fires with `None` if unset.
    pub fn optional<T: OptionalConfigValue>(
        name: impl Into<String>,
        description: impl Into<String>,
        callback: impl FnMut(Option<T>) + Send + 'static,
    ) -> Self { /* ... */ }

    /// Presence-only flag. Callback fires with `true` if the flag is present.
    pub fn flag(
        name: impl Into<String>,
        description: impl Into<String>,
        callback: impl FnMut(bool) + Send + 'static,
    ) -> Self { /* ... */ }

    /// Multi-value string list.
    pub fn multi(
        name: impl Into<String>,
        description: impl Into<String>,
        callback: impl FnMut(&[String]) + Send + 'static,
    ) -> Self { /* ... */ }

    /// Process this option before Mir initialization starts.
    pub fn pre_init(self) -> Self { /* ... */ }
}
```

#### FFI Strategy

Configuration option callbacks cross the FFI boundary via a global `Mutex`-guarded
callback registry. Each callback is stored as a typed enum variant indexed by `u32` ID:

```rust
enum ConfigCallback {
    Int(Box<dyn FnMut(i32) + Send>),
    Double(Box<dyn FnMut(f64) + Send>),
    Str(Box<dyn FnMut(String) + Send>),
    Bool(Box<dyn FnMut(bool) + Send>),
    Flag(Box<dyn FnMut(bool) + Send>),
    OptionalInt(Box<dyn FnMut(Option<i32>) + Send>),
    OptionalStr(Box<dyn FnMut(Option<String>) + Send>),
    OptionalBool(Box<dyn FnMut(Option<bool>) + Send>),
    Multi(Box<dyn FnMut(Vec<String>) + Send>),
}
```

A `ConfigOptionDesc` struct is passed across the CXX bridge containing option metadata
(name, description, type tag, defaults, callback ID, pre_init flag). On the C++ side,
`miral::ConfigurationOption` objects are created from these descriptors with callbacks
that dispatch to Rust via exported `rust_config_callback_*` functions.

### CXX Bridge Strategy (miral-sys)

The `miral-sys` crate uses:

- **bindgen** for all C-linkage enums (37 enums from `common.h` + `events/enums.h`)
- **cxx.rs** for C++ class interactions:
  - **Opaque types**: `MirRunner`, `WindowManagerTools`, `Window`, `WindowInfo`, `ApplicationInfo`, `Output`, `Zone`, `WindowSpecification`
  - **Callbacks**: `cxx` fn pointer support + `Box<dyn Fn>` wrappers via C++ helpers
  - **Trait dispatch**: C++ `WindowManagementPolicy` subclass holding `rust::Box<dyn WindowManagementPolicy>` dispatching virtual calls into Rust

### FFI Callback Pattern (trait → C++ virtual dispatch)

```cpp
// C++ side: A policy adapter that dispatches to Rust
class RustWindowManagementPolicy : public miral::WindowManagementPolicy {
    rust::Box<RustPolicyTraitObject> rust_policy;
public:
    auto place_new_window(...) -> WindowSpecification override {
        return rust_policy->place_new_window(...);
    }
    // ... etc for all virtual methods
};
```

## Requirements

1. **Documentation**: Every public API item (functions, structs, enums, traits, methods, modules) must have a rustdoc docstring. Use `#![deny(missing_docs)]` at the crate root to enforce this at compile time.

1. **crates.io publishable**: Both crates must be structured for publication on crates.io:

   - Proper `Cargo.toml` metadata (`description`, `license`, `repository`, `readme`, `keywords`, `categories`)
   - A `README.md` for each crate prominently stating that the **system-installed miral C++ library** (from the Mir project) is a required runtime/build dependency, with instructions for installing it (e.g., `apt install libmiral-dev` on Ubuntu)
   - The `miral-sys` crate's `build.rs` should use `pkg-config` to locate the installed `miral` library and fail with a clear error message if it's not found
   - The `miral` crate depends on `miral-sys` (path dependency for in-tree, version dependency for crates.io)
   - `links = "miral"` in `miral-sys/Cargo.toml` to prevent multiple versions linking the same native library

## Design Decisions

1. **Workspace handles**: Opaque `#[derive(Clone, PartialEq, Eq, Hash)] struct Workspace(u64)` backed by an ID.

1. **Userdata / per-window state**: External storage pattern (Option D). The policy struct owns `HashMap<Application, T>` / `HashMap<Window, T>` for per-application/per-window data. No `userdata()` API exposed on `WindowInfo`/`ApplicationInfo`. `Window` and `Application` implement `Eq + Hash` to enable this.

1. **Thread safety**: Trait requires `Send` but NOT `Sync`. `WindowManagerTools` is `Send + Sync`.

1. **Error handling**: FFI catches C++ exceptions → `Result<T, MirError>`.

1. **Lifetime management**: `&'a WindowInfo` for borrowed references; `Window` is cheaply cloneable.

1. **Configuration options**: Callback-based API with `FnMut(T) + Send + 'static`. Callbacks may fire multiple times as values change at runtime. `Send + 'static` is required because Mir may invoke callbacks on a server/init thread. A sealed `ConfigValue` trait restricts supported types to `i32`, `f64`, `String`, and `bool`. `OptionalConfigValue` further restricts optional variants to `i32`, `String`, and `bool` (matching C++ `mir::optional_value` support). Options are added via a dedicated `MirRunner::add_config_option()` method, not through the `ServerExtension` trait, because they own non-cloneable callback state.

## Notes

- The existing `evdev-rs` platform demonstrates the `cxx.rs` pattern in this codebase.
- bindgen works perfectly for the C-linkage enum headers — no `extern "C"` wrappers needed since the headers are already pure C.
- The `evdev-rs` crate manually defined ~5 enums in CXX bridge blocks; this was appropriate for that small scope but would not scale to 37 enums.
